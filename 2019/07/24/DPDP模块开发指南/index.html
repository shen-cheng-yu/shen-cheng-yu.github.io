模块向框架注册的回调函数

全局数据区初始化

模块使用的全局数据区初始化。
返回指向全局数据区的指针，0（VOS_NULL_PTR）失败。
模块自行分配的数据区，供模块内部使用，全局（所有pipeline，控制线程）可见，通过（待补充接口）获取。

本地数据取初始化

模块使用的本地数据区初始化。
返回指向本地数据区的指针， 0（VOS_NULL_PTR）失败。
模块自行分配的数据区，供模块内部使用，可见范围为当前pipeline，通过（待补充接口）获取。

配置下发回调

dpdk控制线程在收到对应模块的配置下发后回调。
pCfg是配置内容，cJSON结构体指针。
返回VOS_OK则成功，VOS_ERR 失败。

收到对应模块的配置消息后在DPDK控制线程中调用。若需要下发到各个pipeline的本地数据区，需要模块在此函数中自行发送到pipeline的消息，pipeline收到后调用消息处理回调实现。

命令下发回调

dpdk控制线程在收到对应模块的命令下发后回调。
pstJsonAction是配置内容的cJSON结构体指针，pstJsonReply是应答内容的cJSON结构体指针，已由框架分配，供模块在其中加入应答内容。
VOS_OK成功，VOS_ERR失败。

收到对应模块的命令消息后在DPDK控制线程中调用。由模块自行构造应答消息。

返回VOS_OK表示成功生成应答内容，框架会将应答内容返回给网管平台。VOS_ERR表示无法生成应答内容，框架会自行应答ERR，不会使用pstJsonReply内容。

消息处理回调

pipeline在收到对应模块的消息后回调。VOS_OK成功，VOS_ERR失败。收到对应模块的消息后在pipeline所属的线程中调用。

增加新模块test

修改vgp_module_def.h

主要是增加新模块的枚举定义，将MODULE_ID_TEST放到MODULE_ID_MAX前面就好。

修改vgp_module_def.c

主要是添加新模块的头文件 #include "test.h"，以及添加初始化函数的函数指针g_astModuleInfo[MODULE_ID_TEST].pfModuleInit = test_init;。

完成test.h

头文件里面主要就是添加初始化函数的声明即可。

完成test.c

源程序里面主要就是实现刚刚声明的初始化函数。

初始化函数里面，只有一行打印输出，用来调试。

编写模块的Makefile

这个makefile与test.c，test.h一起放在test文件夹下。test目录放在dpdk目录下。

修改dpdk的Makefile

dpdk目录下有一个大的Makefile，嵌套引用各个下级模块的Makefile，在这个文件里面添加一行对于test模块Makefile的引用。

拷贝test目录

执行quick.sh编译的时候，报错/opt/iot/SmartSecuri-Platform/trunk/app/src/fwd/test/Makefile出错。因为之前将test目录放在dpdk下面，而platform里面的fwd没有test目录，所以需要执行一次拷贝工作。cp -r test /opt/iot/SmartSecuri-Platform/trunk/app/src/fwd/。

修改module.info

vim /SPG/bin/fwd/module.info添加自己增加的新模块。

注意，这里模块前面的序号并不是之前module_id_e中的模块ID。

修改start_fwd.sh

这个脚本里面有个函数start_fwd，决定程序运行的结果是输出到哪里。
默认情况下是将输出重定向到一个文件中去了，所以运行时在终端看不到信息输出。将这个函数的第五行修改为./$APP_NAME -c $mask -- -s $FWD_CLI_FILE_NAME &，这样就是将运行结果重定向到标准输出中，即电脑屏幕。
这样，就能在屏幕上看到新模块输出语句的输出了。
